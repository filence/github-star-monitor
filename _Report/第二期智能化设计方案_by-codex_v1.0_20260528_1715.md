# GitHub Release Telegram 第二期智能化设计方案

> 标题：GitHub Release Telegram 第二期智能化设计方案
> 
> 生成时间：2026-05-28 17:15
> 
> 生成者：Codex
> 
> 版本：v1.0
> 
> 用途：明确第二期接入 DeepSeek 的智能摘要、翻译、总览概括、降级和扩展设计，供后续实施使用

## 1. 背景与目标

当前项目第一期已经具备稳定能力：

- 每日拉取 GitHub 星标仓库 Release
- 汇总今日有更新的仓库
- 发送 1 条总览消息
- 为每个有更新的仓库发送 1 条详情消息
- 在 GitHub Actions 中持久化状态，避免重复补发历史 Release

第二期目标不是推翻第一期，而是在现有稳定链路中加入一层可失败、可降级的智能增强能力：

- 自动把英文 Release 标题翻译成中文
- 自动把 Release Notes 总结成中文
- 自动给总览消息补一段“今天这些更新主要在做什么”的中文概括
- 当模型调用失败时，自动回退到第一期原始消息格式，保证 Telegram 发送成功
- 结构上预留其他模型提供商接口，但当前先只接 DeepSeek API

## 2. 本期范围

### 2.1 本期必须实现

- 接入 DeepSeek 作为默认 LLM 提供商
- 支持总览消息单独做一次智能总结
- 支持每个仓库单独做一次智能处理
- 每个 Release 输出中文标题、中文摘要、关键点
- 总览消息显示中文概括
- Telegram 消息增加图标头部
- 模型失败后自动降级回第一期格式
- 保持第一期 state 更新逻辑稳定

### 2.2 本期明确不做

- 不做多模型实际接入
- 不做复杂缓存系统
- 不做数据库持久化
- 不做成本统计面板
- 不做人工审核流程
- 不做字段级半智能降级

## 3. 总体方案

第二期沿用第一期主链路：

`GitHub 数据抓取 -> 更新识别 -> 格式化消息 -> Telegram 发送 -> state 保存`

在“更新识别”和“格式化消息”之间插入“AI 增强层”：

`GitHub 数据抓取 -> 更新识别 -> AI 增强（可失败） -> 格式化消息 -> Telegram 发送 -> state 保存`

核心原则如下：

- 第一优先级是“消息每天稳定发出去”
- AI 是增强层，不是主链路依赖
- 任何 AI 失败都不能阻断 Telegram 发送
- 任何 AI 失败都不能破坏 state 持久化
- 任何 AI 输出都必须经过结构校验，不可信任模型自由文本

## 4. 模块设计

### 4.1 `src/config.js`

职责：

- 统一读取所有环境变量
- 区分基础运行配置和 LLM 配置
- 在缺少必要配置时给出清晰错误

建议读取的环境变量：

- `GH_STAR_MONITOR_TOKEN`
- `TELEGRAM_BOT_TOKEN`
- `TELEGRAM_CHAT_ID`
- `SEND_EMPTY_SUMMARY`
- `LLM_ENABLED`
- `LLM_PROVIDER`
- `LLM_TIMEOUT_MS`
- `LLM_TEMPERATURE`
- `LLM_MAX_NOTES_CHARS`
- `DEEPSEEK_API_KEY`
- `DEEPSEEK_BASE_URL`
- `DEEPSEEK_MODEL`

### 4.2 `src/prompts.js`

职责：

- 集中管理提示词
- 区分总览总结提示词和仓库详情提示词
- 明确输出必须为 JSON
- 明确长度约束和字段约束

### 4.3 `src/ai/provider.js`

职责：

- 定义统一 provider 接口
- 根据 `LLM_PROVIDER` 选择具体实现
- 当前支持 `deepseek`
- 未来可扩展为 `openai`、`openrouter` 等

### 4.4 `src/ai/deepseek.js`

职责：

- 实现对 DeepSeek Chat Completions API 的调用
- 负责超时控制
- 负责重试策略
- 负责提取模型文本输出

### 4.5 `src/ai/service.js`

职责：

- 准备喂给模型的输入内容
- 对 Release Notes 做截断
- 发起总览级和仓库级调用
- 解析 JSON 结果
- 校验返回字段结构
- 在失败时返回降级结果

这是第二期最核心的业务编排层。

### 4.6 `src/format.js`

职责升级为：

- 优先渲染 AI 增强后的字段
- 如果没有 AI 数据，则渲染第一期原始字段
- 继续做 Telegram 消息长度保护
- 增加头部图标和更友好的中文呈现

### 4.7 `src/main.js`

职责调整为：

- 收集 updates 后调用 AI service
- 将 AI 增强结果交给格式化层
- 发送消息后继续沿用当前 state 更新策略

## 5. 数据结构设计

### 5.1 总览增强结果

```json
{
  "summary_zh": "今天的更新主要集中在依赖升级、Bug 修复和发布流程改进。"
}
```

说明：

- `summary_zh` 为 1 到 3 句中文概括
- 如果总览增强失败，则不返回该字段，直接回退到第一期总览

### 5.2 仓库增强结果

```json
{
  "full_name": "owner/repo",
  "repo_summary_zh": "这个仓库本次更新主要聚焦于稳定性修复和工具链改进。",
  "releases": [
    {
      "release_id": 123,
      "title_zh": "修复 Windows 构建问题的稳定版本",
      "summary_zh": "本次发布主要修复了 Windows 环境下的构建失败，并同步升级了关键依赖。",
      "highlights_zh": [
        "修复构建脚本兼容性问题",
        "升级核心依赖版本",
        "优化日志输出"
      ]
    }
  ]
}
```

说明：

- 用 `release_id` 对齐原始 Release
- `repo_summary_zh` 是仓库级一句话概括
- `title_zh`、`summary_zh`、`highlights_zh` 都属于第二期新增增强字段
- 如果任一关键结构缺失，则该仓库整体降级

## 6. 消息格式设计

### 6.1 总览消息

推荐样式：

- 标题前加 `📣`
- 保留第一期的统计字段
- 新增一段“今日更新概括”
- 保留“今日有更新的仓库”列表

示例结构：

```text
📣 GitHub 星标 Release 日报
生成时间：...
更新仓库数：...
新 Release 数：...
新增纳入跟踪：...

今日更新概括：
今天的更新主要集中在依赖升级、Bug 修复和 CLI 体验优化。

今日有更新的仓库
• owner/repo
• ...
```

### 6.2 仓库详情消息

推荐样式：

- 标题前加 `🚀`
- 先展示仓库级一句话总结
- 每个 Release 同时展示原始标题和中文标题
- 中文摘要放在主体位置
- 关键点列表控制在 2 到 4 条

示例结构：

```text
🚀 owner/repo
仓库链接：...
本次新增 Release：2
本仓库更新概括：...

• 原始标题：v1.2.3
• 中文标题：修复 Windows 构建问题的稳定版本
发布时间：...
Pre-release：否
链接：...
中文摘要：
...
关键点：
- ...
- ...
```

## 7. 降级策略

这是本期最重要的稳定性约束。

### 7.1 总览降级

如果总览模型调用出现下列任一问题：

- 超时
- 鉴权失败
- 429 限流
- 5xx 错误
- 返回非 JSON
- JSON 缺少 `summary_zh`

则：

- 直接使用第一期总览格式
- 不影响后续仓库详情发送

### 7.2 仓库详情降级

如果单个仓库模型调用出现下列任一问题：

- 超时
- 鉴权失败
- 429 限流
- 5xx 错误
- 返回非 JSON
- JSON 缺少 `full_name`
- JSON 中 `releases` 数组缺少对应 `release_id`
- 某个 Release 缺少 `title_zh` 或 `summary_zh`

则：

- 该仓库整条消息回退到第一期格式
- 不做字段级拼装
- 不影响其他仓库消息发送

### 7.3 state 处理

- 不论 AI 成功或失败，只要 Telegram 成功发送了对应消息，就照常更新 state
- 避免因为 AI 降级导致重复通知

## 8. DeepSeek 接入策略

### 8.1 提供商策略

- 当前默认 `LLM_PROVIDER=deepseek`
- 未来 provider 工厂可再接其他模型
- 主流程不直接感知 DeepSeek 细节

### 8.2 调用粒度

用户已确认采用：

- 总览单独调用一次
- 每个仓库单独调用一次

不采用：

- 每个 Release 单独调用

原因：

- 更适合当前“总览 1 条 + 每仓库 1 条”的消息结构
- 同一仓库多个 Release 可共享上下文
- 成本比逐 Release 调用更可控

### 8.3 超时与重试

建议：

- 单次超时 `20000ms`
- 对网络错误、429、5xx 最多重试 1 次
- 对非 JSON、结构缺失不重试，直接降级

## 9. 长度与成本控制

### 9.1 输入裁剪

- 每条 Release Notes 预处理后截断
- 默认每条最多保留约 `3000` 到 `4000` 字符
- 单仓库总输入正文不超过 `LLM_MAX_NOTES_CHARS`

### 9.2 输出约束

提示词中明确要求：

- 中文标题尽量 1 行
- 中文摘要控制在 `80` 到 `140` 字
- 关键点最多 `3` 条
- 总览概括控制在 `1` 到 `3` 句

### 9.3 Telegram 安全裁剪

- 保留第一期的消息长度保护
- 第二期新增 AI 字段也要分别限长
- 超长时给出省略提示，不让消息超过 Telegram 上限

## 10. 测试与验证策略

当前项目未引入正式测试框架，因此本期采用“脚本检查 + 本地模拟 + 失败回归”的方式验证。

### 10.1 配置验证

确认以下场景报错清晰：

- `LLM_PROVIDER` 未设置或不支持
- `DEEPSEEK_API_KEY` 缺失
- `LLM_TIMEOUT_MS` 非数字

### 10.2 成功路径验证

确认：

- 总览出现中文概括
- 仓库详情出现中文标题、中文摘要、关键点
- 头部图标正常显示

### 10.3 失败降级验证

模拟：

- API 超时
- 401
- 429
- 5xx
- 非法 JSON

确认：

- 主流程不中断
- 自动回退到第一期格式
- Telegram 仍可发送

### 10.4 超长文本验证

使用超长 Release Notes 样本确认：

- 输入被裁剪
- 输出被裁剪
- Telegram 消息长度安全

### 10.5 GitHub Actions 验证

手动触发 workflow，确认日志能明确区分：

- GitHub 抓取阶段
- AI 增强阶段
- 降级阶段
- Telegram 发送阶段
- state 保存阶段

## 11. 实施顺序建议

建议按以下顺序推进：

1. 抽离统一配置读取
2. 建立 prompts 与 provider 抽象
3. 实现 DeepSeek provider
4. 实现 AI service 的结构校验和降级
5. 升级格式化层
6. 改造主流程接入 AI
7. 更新 README
8. 本地与 GitHub Actions 验证

## 12. 结论

本方案是对第一期稳定链路的增量增强：

- 有 AI 时，日报更像“中文可读摘要”
- 没 AI 时，仍然退回第一期可用状态
- 结构上为未来接入其他模型提供商预留了接口

这使得第二期可以在可控风险下快速上线，并为后续多模型接入打好基础。
